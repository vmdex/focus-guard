# Focus Guard Clock Tech Decision

## Рішення

Focus Guard Clock v0.1 будемо розробляти як сучасний Windows-застосунок на:

- .NET;
- WinUI 3;
- Windows App SDK;
- MSIX packaging у перспективі.

## Чому WinUI 3

WinUI 3 обрано як сучасний варіант для Windows-застосунку, тому що проєкт має розвиватися не тільки як швидкий локальний таймер, а як основа майбутнього polished Windows client для системи Focus Guard.

Причини:

- сучасний Windows 11 вигляд;
- нативні UI-компоненти;
- природніша інтеграція з modern Windows app model;
- кращий шлях до MSIX-пакування;
- перспективність для публікації або нормального встановлення;
- краще відповідає майбутньому баченню платформи Focus Guard.

## Що приймаємо як trade-off

WinUI 3 може бути складнішим на старті, ніж WPF.

Можливі складнощі:

- більше налаштувань проєкту;
- складніше packaging;
- більше уваги до app lifecycle;
- потенційно більше нюансів зі сповіщеннями;
- складніше, ніж простий portable `.exe`.

Цей trade-off прийнятий свідомо, бо ціль проєкту - не тільки швидкий експеримент, а фундамент для сучасного Windows-клієнта.

## Альтернативи

### WPF

Плюси:

- швидкий старт;
- простий desktop app;
- багато документації;
- легше дебажити;
- достатньо для MVP.

Мінуси:

- менш сучасний вигляд за замовчуванням;
- слабше відповідає modern Windows app model;
- не так природно лягає в майбутній MSIX-first підхід.

### Electron

Плюси:

- швидкий UI на web-технологіях;
- багато готових бібліотек.

Мінуси:

- занадто важкий для простого timer app;
- більший runtime;
- менше відчуття нативного Windows-застосунку.

### Tauri

Плюси:

- легший за Electron;
- сучасний підхід;
- web UI.

Мінуси:

- складніший стек;
- Rust layer додає зайву складність для першого Windows MVP;
- нативні Windows-можливості можуть вимагати більше інтеграційної роботи.

### Avalonia

Плюси:

- .NET;
- кросплатформеність;
- хороша альтернатива WPF.

Мінуси:

- кросплатформеність поки не є головною вимогою;
- WinUI 3 краще відповідає саме modern Windows-напрямку.

## Початкова архітектура

Проєкт бажано тримати так, щоб логіка таймера не була прив'язана напряму до UI.

Пропонована структура майбутнього app:

```text
projects/focus-guard-clock/
  app/
    FocusGuard.Clock.App/
      Views/
      ViewModels/
      Services/
      Models/
      App.xaml
      MainWindow.xaml
    FocusGuard.Clock.Core/
      Models/
      Services/
      TimerEngine/
    FocusGuard.Clock.Tests/
```

## Принципи реалізації

- UI не має містити основну бізнес-логіку таймера.
- Розрахунок focus/break cycle має бути в окремому core-шарі.
- Локальні settings і history мають бути заховані за сервісами.
- Звуки і сповіщення мають бути окремими сервісами.
- Перший прототип може бути простішим, але структура не має заважати майбутньому backend/client поділу.

## Перший технічний крок

Наступний крок - перевірити локальне середовище:

- чи встановлено .NET SDK;
- які версії SDK доступні;
- чи є шаблони WinUI 3 / Windows App SDK;
- чи можна створити і зібрати мінімальний WinUI 3 проєкт.
